ADASSettingManager API 文档

类 ADASSettingManager

ADASSettingManager 是一个高级驾驶辅助系统(ADAS)设置管理类，用于配置车辆的各种ADAS功能。它采用单例模式设计，通过 getInstance() 方法获取实例。

依赖说明 ：此类依赖于 VehicleTypeManager 和 CanBusClient 类。

方法列表

getInstance()

描述 ：获取 ADASSettingManager 的单例实例。

返回 ： ADASSettingManager 的实例。 

@classmethod
def getInstance(cls)

adassetting_configuration()

描述 ：配置车辆的ADAS功能设置。当前仅支持CLEA平台。

功能列表 ：

• 前置行人预警
• 自动防撞准备
• 车辆通过紧急制动
• 后方自动紧急制动
• 后方车辆通过预警
• 变道预警
• 侧盲区预警
• 开门预警
• 行驶风格
• 车道偏离辅助
• 跟车起步提醒
• 弯道自动限速
• 自动调整车速
• 交通信号与停止线控制
• 智能辅助变道
• 智能避让
• 限速提示
• 智慧领航 
def adassetting_configuration(self)

使用示例

基本使用 

# 获取ADASSettingManager实例
adas_manager = ADASSettingManager.getInstance()

# 配置ADAS设置
adas_manager.adassetting_configuration()

注意事项

  • 此类的功能目前仅支持CLEA平台
  • 调用adassetting_configuration()方法前需要确保CanBusClient已正确连接
  • 所有配置都是通过CAN总线发送信号实现的
  • 每个功能配置会发送多个CAN信号以确保设置生效
  • 实际效果可能因车辆型号和配置不同而有所差异

技术实现

ADASSettingManager 通过以下方式实现功能配置：

  • 使用单例模式确保全局唯一实例
  • 依赖VehicleTypeManager获取车辆平台信息
  • 通过CanBusClient发送CAN信号配置各项功能
  • 每个功能配置包含多个相关信号的设置
  • 使用特定的CAN ID和信号名称进行精确控制

AudioRecoringThread

一个线程类，用于定期录制音频文件。

注意： 类名拼写有误，应为"AudioRecordingThread"

__init__(self, caseFolder)
构造函数，初始化音频录制任务。

  • caseFolder : 测试用例文件夹路径，用于存储录制的音频文件
stopTask(self)
停止音频录制任务。
recordAudio(self)
执行单次音频录制，持续30秒。

实现细节:

  • 使用当前时间生成唯一的音频文件名
  • 调用AudioManager的startRecordAudio方法开始录制
  • 等待30秒后调用stopRecordAudio方法停止录制
run(self)
线程主循环，每隔30分钟执行一次音频录制。

工作流程:

  1. 检查是否收到停止信号
  2. 如果达到时间间隔(30分钟)或首次运行，执行录制
  3. 循环检查直到任务停止

常量

  • TIME_INTERVAL : 音频录制间隔时间(秒)，默认为1800(30分钟)

使用示例 

# 创建音频录制线程
audio_thread = AudioRecoringThread("C:/test_case")
# 启动录制
audio_thread.start()
# 停止录制
audio_thread.stopTask()
audio_thread.join()

BaseLocator 类

该类提供了基于文件路径的指示器获取功能，内置缓存机制以提高性能。

__init__(self)
构造函数，初始化BaseLocator实例。

初始化时会创建一个LFU缓存实例。

getIndicator(self, filePath)
获取指定文件路径的指示器内容。

该方法会首先检查缓存中是否已有该文件内容，如果没有则从文件系统读取并解析JSON内容，然后缓存结果。

参数:
  • filePath - 要读取的文件路径
返回: 解析后的JSON对象，如果失败返回None

缓存行为说明

  • 使用LFU(Least Frequently Used)缓存策略
  • 缓存键为文件路径
  • 缓存值为解析后的JSON对象
  • 当缓存未命中时，会自动从文件系统读取并更新缓存

异常处理

  • 所有异常都会被捕获并记录日志
  • 发生异常时方法会返回None
  • 常见的异常情况包括：文件不存在、文件格式错误、JSON解析失败等

ClusterFunctionThread API 文档

概述

ClusterFunctionThread 是一个继承自 threading.Thread 的类，用于在集群环境中随机触发功能模块的线程。它可以在指定的时间间隔内随机激活集群模块的功能，并支持暂停、恢复和停止操作。

类初始化

__init__(module, sendInterval=SEND_MESSAGE_INTERVAL, duration=0, randomSeed=0)

描述 : 初始化集群功能线程。

参数 :

  • module : 集群模块名称
  • sendInterval : 消息发送间隔(秒)，默认0.3秒(300ms)
  • duration : 任务持续时间(秒)，0表示无限
  • randomSeed : 随机种子，用于可重复的随机行为

主要方法

isTaskTimeout(startTime, duration)

描述 : 检查任务是否超时。

参数 :

  • startTime : 任务开始时间
  • duration : 任务持续时间

返回 : bool 是否超时

stopTask()

描述 : 停止任务并等待线程结束。

pauseTask()

描述 : 暂停任务执行。

resumeTask()

描述 : 恢复暂停的任务。

activateAllFuntions()

描述 : 激活模块的所有功能。

resetAllFuntions()

描述 : 重置模块的所有功能。

run()

描述 : 线程主函数，执行随机触发功能的任务。

行为 :

  • 获取模块的所有功能键
  • 记录开始时间
  • 使用随机种子初始化随机数生成器
  • 循环随机选择功能并激活，直到任务停止或超时
  • 支持暂停/恢复功能

常量

常量名 值 描述
SEND_MESSAGE_INTERVAL 0.3 默认消息发送间隔(300毫秒)

使用示例 


# 创建集群功能线程
cluster_thread = ClusterFunctionThread(
    module="example_module",
    sendInterval=0.5,  # 500ms间隔
    duration=60,       # 运行60秒
    randomSeed=42      # 固定随机种子
)

# 启动线程
cluster_thread.start()

# 暂停任务
cluster_thread.pauseTask()

# 恢复任务
cluster_thread.resumeTask()

# 停止任务
cluster_thread.stopTask()

工作原理

ClusterFunctionThread 通过以下方式工作:

  1. 初始化时设置模块、发送间隔、持续时间和随机种子
  2. 启动线程后，随机选择模块的功能键并发送激活消息
  3. 每次发送后等待指定的间隔时间
  4. 可以随时暂停或恢复任务
  5. 当达到指定持续时间或调用stopTask()时，线程结束

注意事项

  • 线程启动后会自动运行，不需要手动调用run()方法
  • 设置randomSeed可以确保每次运行产生相同的随机序列
  • duration=0表示线程会一直运行直到手动停止
  • 暂停状态下线程会每5秒检查一次是否恢复

LifeCycle 模块 API 文档

模块概述

LifeCycle 模块提供了一套装饰器，用于管理测试场景的生命周期，包括初始化(setup)、执行(main)和清理(teardown)三个阶段。该模块集成了日志记录、文件管理和线程监控等功能。

装饰器 API

setup(senario)

测试场景初始化装饰器工厂函数。

参数说明:

  • senario (str): 测试场景名称，用于创建专属日志目录

功能说明:

  1. 在基础日志路径下创建格式为 {senario}_{当前时间} 的日志目录
  2. 执行被装饰函数并传入日志目录路径参数
  3. 捕获并记录执行过程中的所有异常

示例代码: 

@setup("登录测试")
def test_login(*args, logFolder):
    # 测试代码
    pass

main(func)

测试执行阶段核心装饰器。

参数说明:

  • func : 被装饰的测试函数

功能说明:

  1. 执行被装饰的测试函数
  2. 提供空的finally块用于扩展

示例代码: 

@main
def test_functionality():
    # 测试代码
    pass

teardown(func)

测试清理阶段装饰器。

参数说明:

  • func : 被装饰的清理函数

功能说明:

  1. 执行被装饰的清理函数
  2. 等待30秒后执行清理操作
  3. 清理阶段会：
    • 列出所有运行中的线程
    • 如果存在多个线程，强制退出程序( os._exit(0) )
    • 捕获并记录清理过程中的异常

示例代码: 

@teardown
def cleanup():
    # 清理代码
    pass

依赖模块

  • FileUtil : 文件和目录操作工具
  • LogUtil : 日志记录工具
  • TimeUtil : 时间处理工具
  • Constant : 常量定义
  • Python标准库: os , threading , time , functools

典型使用模式

通常组合使用这三个装饰器来管理完整的测试场景生命周期： 

@setup("用户登录场景")
@main
@teardown

    # 测试实现代码
    pass

注意事项

  • setup 装饰器会自动为每个场景创建带时间戳的日志目录
  • teardown 装饰器包含强制清理逻辑，当存在未结束线程时会强制终止程序
  • 所有异常都会被捕获并记录在整个生命周期过程中
  • 清理阶段有30秒延迟，确保测试结果完全输出

register(function)

注册函数并返回对应的请求和回调。

参数说明:

参数名 类型 说明
function str 要注册的函数名

返回值:

tuple - (request, callback) 请求对象和回调函数，如果出错则返回None

功能说明:

  1. 检查注册表中是否存在指定函数的"callback"键
  2. 从注册表获取request和callback名称
  3. 通过_instance获取回调函数对象
  4. 返回(request, callback)元组
  5. 捕获并打印所有异常

示例代码: 

# 注册函数
request, callback = register.register("func1")

# 使用返回的回调
result = callback(request)

注意事项

  • 该类使用单例模式(_instance类变量)
  • 注册表(table)应为字典结构，包含"callback"和"request"信息
  • register方法会打印异常但不会抛出，调用方需检查返回值
  • 回调函数需要通过_instance对象获取

    }

register = Register(reg_table)

# 注册并获取回调

    # 使用回调处理请求

SystemServiceChecker API 文档

概述

SystemServiceChecker 是一个继承自 threading.Thread 的系统服务检测线程类，用于监控关键系统服务(com.android.car和com.patac.service)的运行状态，当检测到服务崩溃时自动收集相关日志。

类初始化

__init__(caseFolder)

描述 : 初始化系统服务检测线程。

参数 :

  • caseFolder : 日志保存的基础目录路径

常量

常量名 值 描述
GET_ANDROID_CAR_PID_CMD adb shell命令 获取com.android.car进程ID的命令
GET_PATAC_SERVICE_PID_CMD adb shell命令 获取com.patac.service进程ID的命令
CHECK_ANDROID_CAR_PID_DURATION 1800 (30分钟) com.android.car进程检查周期
CHECK_PATAC_SERVICE_PID_DURATION 1800 (30分钟) com.patac.service进程检查周期
ANDROID_CAR_CRASH_TIMES_MAX 5 com.android.car进程崩溃最大允许次数
PATAC_SERVICE_CRASH_TIMES_MAX 5 com.patac.service进程崩溃最大允许次数
EXCEPTION_CHECK_INTERVAL 60 (60秒) 异常检查间隔时间

描述 : 获取当前com.android.car进程ID。

返回 : str 进程ID，获取失败返回None

描述 : 获取当前com.patac.service进程ID。

返回 : str 进程ID，获取失败返回None

描述 : 检查com.android.car进程是否变化。

逻辑 :

  • 每30分钟重置崩溃计数
  • 比较当前进程ID与初始值
  • 进程变化则增加崩溃计数
  • 崩溃次数超过阈值返回False

返回 : bool 进程是否正常

checkPatacService()

描述 : 检查com.patac.service进程是否变化。

逻辑 :

  • 每30分钟重置崩溃计数
  • 比较当前进程ID与初始值
  • 进程变化则增加崩溃计数
  • 崩溃次数超过阈值返回False

返回 : bool 进程是否正常

描述 : com.android.car崩溃处理函数，收集以下日志:

  • Tombstones日志(/data/tombstones)
  • Gmlogger日志(/data/gmlogger)
  • Dropbox日志(/data/system/dropbox)
  • 系统dump(adb shell dumpsys)
onPatacServiceCrashed()

描述 : com.patac.service崩溃处理函数，收集日志(同onAndroidCarCrashed)。

run()

描述 : 线程主函数，执行服务检测循环。

检测逻辑 :

  1. 获取初始进程ID
  2. 循环检测(间隔60秒):
    • 检查com.android.car进程
    • 检查com.patac.service进程
  3. 任何一项检查失败则收集相关日志
stopTask()

描述 : 停止服务检测任务。

使用示例 


# 创建系统服务检测线程
service_checker = SystemServiceChecker(
    caseFolder="/path/to/log/folder"
)

# 启动线程
service_checker.start()

# 停止线程
service_checker.stopTask()

检测流程

  1. 初始化时设置日志保存目录
  2. 启动线程后获取初始进程ID
  3. 周期性检测两个关键服务:
    • com.android.car
    • com.patac.service
  4. 检测到服务重启超过阈值(5次/30分钟)则判定为崩溃
  5. 崩溃时自动收集系统日志并保存到指定目录
  6. 调用stopTask()可以停止检测

注意事项

  • 检测间隔为60秒，但内部循环间隔为5秒
  • 崩溃计数每30分钟重置一次
  • 日志收集操作可能需要较长时间(每个pull操作默认5分钟超时)
  • dumpsys操作可能需要更长时间(5分钟超时)
  • 需要确保ADB连接正常

TcpDumper API 文档

该模块提供了同时收集IVI(车载信息娱乐系统)和QNX系统的TCP网络数据包的功能，支持定时自动停止和手动停止。

从IVI系统拉取捕获的数据包文件。

删除IVI系统上的数据包文件。
